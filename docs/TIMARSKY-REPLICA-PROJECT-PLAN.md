# Timarsky 复刻项目计划详情

> 文档日期：2026-04-11
> 项目目标：基于当前 `shuang-qian` 框架复刻 `www.timarsky.com` 的核心产品能力与落地页体验
> 复刻策略：先完成可上线 MVP，再逐步补齐数字人、提示词优化、作品资产与商业化闭环

---

## 1. 项目目标

本项目不是单纯做一个官网，而是要复刻 Timarsky 的核心业务闭环：

1. 用户进入官网或工作台后，能快速理解产品价值。
2. 用户可以通过一句话、上传图片或上传人物素材，发起 AI 图片/视频生成。
3. 系统能够自动优化 Prompt，降低普通用户使用门槛。
4. 系统具备任务排队、积分扣费、历史记录、作品下载、代理分销的完整业务流程。
5. 整体实现基于当前仓库现有技术栈完成，不新起一套独立后端。

本次建议的复刻范围分为两层：

- 第一层：复刻 Timarsky 的视觉表达、信息架构与核心转化路径。
- 第二层：复刻 Timarsky 的 AI 创作能力，重点做图生视频、文生视频、数字人分身、Prompt 优化。

---

## 2. 当前项目基础与复用策略

### 2.1 现有项目可复用部分

当前仓库已经具备较好的业务底座：

- 前端：当前使用 `shuang-ui/zhuimeng-dream`，技术栈为 `Vue 3 + TypeScript + Vite`
- 后端：`Spring Boot 3.5.9 + MyBatis Plus + Redis + MySQL`
- 模块化能力：
  - `shuang-module-ai`：AI 相关业务扩展入口
  - `shuang-module-member`：用户体系
  - `shuang-module-pay`：支付与积分体系可扩展
  - `shuang-module-agency`：代理与分销体系
  - `shuang-module-infra`：文件、存储、基础设施能力可扩展

### 2.2 复刻原则

- 不推翻现有框架，直接在当前仓库上扩展。
- 官网展示页和工作台统一由 `shuang-ui/zhuimeng-dream` 承接。
- AI 任务、积分、历史记录、资产库、支付全部挂到 Spring Boot 业务模块。
- 大模型采用“双供应商接入”策略：主通道优先阿里百炼，备通道为豆包。
- 视频生成采用“先接商用 API，再评估自建模型”的落地路径，避免前期过重投入 GPU 基础设施。

## 4. 技术方案总览

### 4.1 总体架构

前端层：

- `shuang-ui/zhuimeng-dream` 负责官网、创作台、资产库、账户中心、支付页
- 使用 Vue Router 组织页面
- 前端直接调用 Spring Boot API

后端层：

- `shuang-module-ai`：AI 任务编排、Prompt 优化、模型供应商适配
- `shuang-module-member`：用户登录态、个人资料、权限
- `shuang-module-pay`：积分钱包、扣费、充值、订单
- `shuang-module-agency`：邀请、返佣、推广统计
- `shuang-module-infra`：素材上传、对象存储、任务日志、回调处理

基础设施：

- MySQL：用户、任务、作品、资产、订单、积分流水
- Redis：任务队列、状态缓存、限流、幂等控制
- OSS/S3：图片、视频、封面、数字人素材
- 可选消息队列：先用 Redis Stream 或现有异步能力，后期再升级 MQ

### 4.2 模型接入策略

#### Prompt 大模型

主方案：

- 阿里百炼

备方案：

- 豆包大模型

用途：

- Prompt 扩写
- Prompt 翻译
- 视频脚本润色
- 行业模板改写
- 失败结果二次重写
- 生成前复杂度评估与点数预估

#### 视频生成模型

优先接商用 API：

- 可灵/Kling
- Runway
- Pika
- 即梦或其他稳定商用视频接口

数字人能力优先接成熟 API：

- HeyGen
- D-ID

说明：

- 首版不建议直接自建 CogVideoX、AnimateAnyone、LivePortrait 全链路，原因是部署复杂、显存成本高、稳定性验证周期长。
- 第一阶段先用外部 API 跑通业务闭环，第二阶段再评估自建推理服务替换高成本环节。

## 6. 技术难点与解决方案

## 6.1 难点一：视频大模型接入成本高、稳定性差

### 问题

- 单次视频生成耗时长，用户等待敏感
- 商用 API 存在配额、速率、失败率问题
- 不同供应商返回结构不同，难以统一编排

### 解决方案

- 后端抽象统一 `AiVideoProvider` 适配层
- 首期至少接入 2 家供应商，主用一套、备份一套
- 所有生成任务统一进入异步任务表，不允许前端直接同步等待
- 建立任务状态机：`CREATED / PENDING / RUNNING / SUCCESS / FAILED / REFUND`
- 对失败任务做有限次自动重试
- 把扣费拆成“预扣费 + 完成结算 + 失败回滚”

### 落地建议

- `shuang-module-ai` 增加 provider 适配器、任务服务、回调服务
- Redis 做任务状态缓存和限流
- 所有第三方调用必须记录 requestId、supplierTaskId、耗时、费用

## 6.2 难点二：人物一致性与数字人效果不稳定

### 问题

- 同一人物在不同视频中容易脸崩、手崩、服装漂移
- 图生视频和数字人驱动对源图质量高度敏感
- 用户上传的照片质量参差不齐

### 解决方案

- 上传阶段增加素材校验：清晰度、正脸检测、遮挡检测
- 对数字人素材做预处理：裁剪、增强、背景清理、关键点检测
- 数字人首版优先调用成熟 API，避免自建链路稳定性问题
- 建立“推荐素材规范”与失败原因提示，减少无效任务

### 落地建议

- 增加素材质量评分接口
- 分身功能默认限制首版时长和镜头复杂度
- 对生成结果做人工验收后台或抽检机制

## 6.3 难点三：Prompt 工程决定最终效果

### 问题

- 普通用户不会写 Prompt
- 同一句话交给视频模型，结果波动非常大
- 视频 Prompt 不只是描述主体，还包含镜头、动作、节奏、光影和风格

### 解决方案

- 建 Prompt 中间层，不让用户原始输入直接进视频模型
- 使用阿里百炼作为主 Prompt 优化引擎，豆包作为备份或 AB 测试通道
- 构建结构化 Prompt Schema，而不是纯文本拼接
- 做行业模板库、风格模板库、镜头模板库
- 对失败结果提供“一键重写 Prompt 再试一次”

### 推荐 Prompt 流程

1. 用户输入一句话
2. LLM 识别行业、目标、主体、风格
3. LLM 输出结构化字段
4. 模板引擎拼装成对应供应商需要的 Prompt
5. 根据模型差异自动裁剪长度和关键词

### 为什么这一块是重中之重

用户不会感知你后端用了什么框架，但会立刻感知“输入一句话到底能不能出片”。所以 Prompt 层不是辅助模块，而是产品灵魂，必须单独排期、单独调优、单独验收。

## 6.4 难点四：异步任务、积分扣费、结果回调必须一致

### 问题

- 用户并发提交时容易重复扣费
- 第三方回调晚到、重复回调、失败回调都很常见
- 作品生成成功但数据库状态没更新，会造成投诉

### 解决方案

- 所有任务创建接口增加幂等键
- 积分流水、任务表、订单表使用事务控制
- 第三方回调做签名校验和幂等落库
- 定时补偿任务扫描超时任务
- 失败任务自动退款并写明原因

## 6.5 难点五：官网看起来简单，实际转化链路很长

### 问题

- 如果只有漂亮首页，没有案例、流程、价格、入口，转化很差
- 如果功能很多但信息架构混乱，用户容易流失

### 解决方案

- 首页只讲三个核心价值：快速出片、低门槛、营销转化
- 首页 CTA 固定指向“立即生成”和“查看案例”
- 案例库、套餐页、下载中心、工作台入口需要形成闭环
- 工作台内保持最短操作路径：上传/输入 -> 优化 Prompt -> 生成 -> 下载

## 7. 阿里百炼 + 豆包双通道方案

### 7.1 目标

保证 Prompt 优化与 AI 文案辅助有双供应商可切换能力，避免单点依赖。

### 7.2 设计建议

接口抽象：

- `PromptOptimizeService`
- `PromptProvider`
- `AlibabaBailianPromptProvider`
- `DoubaoPromptProvider`

路由策略：

- 默认走阿里百炼
- 阿里失败时自动切到豆包
- 后台可配置不同场景使用不同供应商
- 支持后续按成本、响应速度、效果做 AB Test

适用场景：

- 视频 Prompt 扩写
- 图片 Prompt 优化
- 文案润色
- 行业模板改写
- 多语言翻译

建议配置项：

- 是否启用主备切换
- 单次请求超时时间
- 最大重试次数
- 每日调用限额
- 按场景指定默认供应商

## 11. 验收标准

MVP 阶段建议用以下标准验收：

1. 用户可注册登录并进入工作台。
2. 用户可输入一句话完成 Prompt 优化并创建视频任务。
3. 用户可上传图片完成图生视频任务。
4. 任务结果可正确回写并进入作品库。
5. 积分扣费、失败退款、历史记录查询全部可用。
6. 官网首页、案例页、套餐页、下载中心可正常访问。
7. 阿里百炼与豆包至少一主一备可切换。

## 12. 风险与建议

### 主要风险

- 商用视频 API 不稳定或价格波动
- Prompt 调优周期超预期
- 数字人效果不达标导致用户落差
- 第三方回调链路不稳定
- 涉及人像与 deepfake 的审核风险

### 建议

- 第一版优先上线视频生成闭环，不要一开始就做过重的自建推理服务
- Prompt 系统必须尽早开始调优，不要等页面完成后再补
- 数字人功能分两期做，先接成熟 API，再评估开源替换
- 账号密码不要写入仓库文档，测试账号只放本地安全配置

## 13. 最终实施建议

从当前项目出发，最稳妥的路径不是“完全 1:1 技术复制 Timarsky”，而是“在现有 `Spring Boot + Vue/Vite` 架构上，把 Timarsky 的核心用户路径和商业闭环快速复刻出来”。

推荐优先级如下：

1. 先做官网转化页 + 视频生成闭环
2. 再做 Prompt 中台与模板体系
3. 再做数字人分身一期
4. 最后补齐代理分销、下载中心、更多 AI 工具

如果按这个方案执行，6 周内可以完成一个可展示、可试用、可付费的 Timarsky 复刻版 MVP。
