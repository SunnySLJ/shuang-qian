# GStack 使用示例

这份文档给当前项目提供一组可以直接在 Codex / Claude Code 中使用的 `gstack` 示例话术。

适用场景：

- 新功能规划
- 技术方案审查
- Bug 排查
- 代码 Review
- 页面 QA
- 发布前检查

## 推荐工作流

日常开发建议按这个顺序使用：

1. `gstack-office-hours`
2. `gstack-autoplan`
3. 实现功能
4. `gstack-review`
5. `gstack-qa` 或 `gstack-qa-only`
6. `gstack-ship`

## 示例 1：先把需求讲清楚

适合你还没完全想明白怎么做，想先把业务边界讲清楚的时候。

你可以这样对 Codex 说：

```text
加载 gstack。使用 gstack-office-hours。

我要做“爆款拆解”功能，目标是让用户上传视频链接或文案后，
系统自动提取结构、卖点、镜头节奏和转化话术，
最后生成一个适合二创的拆解结果页。

请先不要写代码，先帮我把下面几件事问清楚：
1. 真实用户是谁
2. 这个功能和现有 AI 生图/生视频/代理分销的关系
3. 最小可上线版本应该砍掉哪些内容
4. 需要哪些数据表和任务链路
5. 哪些地方最容易做成伪需求
```

预期输出：

- 功能目标是否成立
- MVP 范围
- 核心输入输出
- 风险和依赖项

## 示例 2：自动生成一份完整规划

适合新功能正式开做前，先把产品、工程、设计审一遍。

```text
加载 gstack。使用 gstack-autoplan。

基于当前项目，规划一个“AI 视频一键拆解”能力：
- 前端入口放在现有 AI 视频相关页面
- 支持用户输入视频链接
- 后端异步调用模型做文案、分镜、卖点和镜头节奏分析
- 分析结果要可保存、可查看历史
- 每次拆解消耗积分
- 失败时要退款或补偿

输出要求：
1. 给出最小可交付方案
2. 明确涉及的模块、表、接口、异步任务
3. 明确积分扣减、幂等、失败补偿怎么做
4. 输出后不要直接编码，先给我方案结论
```

预期输出：

- 产品方案
- 工程拆分
- 风险点
- 推荐实施顺序

## 示例 3：只做技术方案审查

适合你已经有想法，只需要从工程侧挑刺。

```text
加载 gstack。使用 gstack-plan-eng-review。

请审查这个方案：
我准备在 `shuang-module-ai` 增加视频拆解服务，
Controller 放在 `controller.app`，
Service 内部先扣积分，再创建拆解记录，再投递异步任务，
异步结果回填到数据库，前端轮询结果。

重点帮我看：
1. 模块边界是否合理
2. 扣积分和异步执行是否有一致性问题
3. 幂等设计是否足够
4. 高并发下会不会重复扣费
5. 需要补哪些测试
```

建议补充上下文：

- 相关表结构
- 已有钱包扣减逻辑
- 目标接口草案

## 示例 4：让它先做问题排查，不要盲修

适合线上或联调时遇到异常。

```text
加载 gstack。使用 gstack-investigate。

现象：
部分用户发起 AI 视频生成后，积分已经扣了，
但生成记录状态一直停留在处理中，最终也没有回滚积分。

请先排查，不要直接修改代码。
重点检查：
1. 扣积分事务和生成记录创建是否分离
2. 异步任务失败后有没有补偿
3. 是否存在第三方回调丢失
4. 是否存在重复请求导致状态错乱
5. 给出最可能的 3 个根因和验证方法
```

适用位置：

- 视频生成卡死
- 充值成功但积分没到账
- 代理分佣不正确

## 示例 5：对当前改动做代码审查

适合你已经写完代码，准备让它找风险。

```text
加载 gstack。使用 gstack-review。

请 review 当前分支改动，重点按生产风险排序输出问题。
项目背景：
- Spring Boot 多模块 Maven
- 涉及积分、AI 生成、代理分销
- 对事务、幂等、审计要求很高

请重点关注：
1. 扣积分链路
2. 退款/回滚逻辑
3. Mapper 和 XML 查询是否有遗漏
4. 接口参数校验和权限控制
5. 是否缺关键测试
```

适合在这些改动后运行：

- 积分扣减相关
- 支付回调相关
- 代理分佣相关
- AI 生成任务相关

## 示例 6：用浏览器做真实 QA

适合页面已经能访问，需要实际点一遍流程。

```text
加载 gstack。使用 gstack-qa。

请测试本地页面 `http://localhost:8888/ai/video/analyze`。
重点验证：
1. 输入视频链接后能否正常提交
2. 积分不足时提示是否清晰
3. 提交后 loading、轮询、失败提示是否合理
4. 成功后结果页是否展示完整
5. 控制台是否有报错

如果发现明显问题，先修复再回归验证。
```

如果你只想拿报告，不让它改代码：

```text
加载 gstack。使用 gstack-qa-only。

请测试本地页面 `http://localhost:8888/ai/video/analyze`，
只输出问题报告，不要修改代码。
```

## 示例 7：发布前做一次完整检查

适合功能完成后准备提 PR 或发布。

```text
加载 gstack。使用 gstack-ship。

请为当前分支做发布前检查：
1. 跑必要测试
2. 检查本次改动有没有遗漏文档
3. 检查数据库变更是否同步
4. 检查 AI、支付、积分、代理相关高风险逻辑
5. 总结这次改动适不适合发 PR
```

适合搭配：

- `gstack-review`
- `gstack-document-release`

## 示例 8：做一次安全审计

适合上线前或接入第三方支付、回调、上传能力后。

```text
加载 gstack。使用 gstack-cso。

请对当前项目做安全审计，重点看：
1. 支付回调接口
2. 文件上传和外部 URL 输入
3. 代理邀请码和绑定逻辑
4. 权限绕过风险
5. 敏感配置泄露风险

请按严重级别输出，并尽量给出可利用场景。
```

## 示例 9：复盘最近一轮开发

适合每完成一个批次后复盘。

```text
加载 gstack。使用 gstack-retro。

请基于当前仓库最近一段时间改动做一次复盘：
1. 哪些功能推进顺利
2. 哪些地方返工最多
3. 哪些 Bug 本来可以更早发现
4. 哪些地方测试覆盖还不够
5. 下一轮开发最该优化什么
```

## 适合本项目的常用组合

### 组合 A：新功能立项

```text
加载 gstack。先运行 gstack-office-hours，再运行 gstack-autoplan。
主题：新做一个 AI 视频拆解功能。
先输出方案，不要直接编码。
```

### 组合 B：修线上问题

```text
加载 gstack。先运行 gstack-investigate，再根据排查结论决定是否运行 gstack-review。
问题：用户生成失败后积分没有退回。
```

### 组合 C：页面联调

```text
加载 gstack。运行 gstack-qa-only。
测试地址：http://localhost:8888/ai/video/analyze
只给我问题列表和复现步骤。
```

### 组合 D：提 PR 前最后一轮

```text
加载 gstack。先运行 gstack-review，再运行 gstack-ship。
```

## 使用建议

- 涉及积分、支付、分佣时，优先用 `gstack-plan-eng-review`、`gstack-review`、`gstack-investigate`
- 涉及页面体验时，优先用 `gstack-qa`、`gstack-qa-only`、`gstack-browse`
- 涉及新功能方向不确定时，优先用 `gstack-office-hours`、`gstack-autoplan`
- 不要把多个目标混成一句空泛需求，尽量明确页面、模块、接口或故障现象
- 如果你只想要方案，不想自动改代码，要在提示里明确写“不要直接编码”

## 一句最实用模板

平时最省事的写法：

```text
加载 gstack。使用 gstack-review。
请基于当前仓库和当前分支改动，按生产风险排序输出问题，重点看事务、幂等、积分回滚、权限校验和测试遗漏。
```
